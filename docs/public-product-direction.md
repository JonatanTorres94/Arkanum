# Arkanum — Public Product & UX Direction

## 1. Propósito del documento

Este documento define la dirección estratégica de producto y UX para el sitio web público de Arkanum.

Debe leerse obligatoriamente antes de abrir cualquier issue de implementación, diseño o copy relacionado con la web pública.

No es un documento de código. No define componentes, rutas ni esquemas. Define el posicionamiento, la narrativa, los principios de diseño y la escalera de soluciones que deben guiar todo el trabajo de UX/diseño futuro.

---

## 2. Posicionamiento central

Arkanum no vende chatbots.

Arkanum construye sistemas operativos inteligentes para empresas.

No toda solución requiere IA. Algunas soluciones deben sentirse inteligentes porque la operación, la arquitectura, el UX y la automatización están bien diseñados. La IA se integra solo cuando agrega valor real: contexto, razonamiento, velocidad, soporte a decisiones, automatización o ejecución.

Principio de trabajo:

```text
Primero arquitectura.
Luego automatización.
Luego IA, cuando realmente incrementa el valor operativo.
```

Posible framing público:

```text
IA conectada a tus procesos, no aislada en una ventana de chat.
```

---

## 3. El enemigo

El enemigo principal son las empresas que crecieron pero siguen operando procesos críticos con herramientas improvisadas.

Posible framing:

```text
Tu empresa creció. Tus sistemas no.
```

Esto incluye:

- planillas como columna vertebral operativa
- WhatsApp como gestor de procesos
- sistemas desconectados
- software legado que no habla con otras herramientas
- automatizaciones frágiles
- reportes manuales
- dependencia de personas clave y memoria humana
- agencias/proveedores que entregan y desaparecen

---

## 4. Balance emocional deseado

El visitante que abandona el sitio debe pensar:

```text
40% — Estas personas saben lo que hacen.
35% — Quiero algo así para mi empresa.
25% — Esto se siente como el futuro de los sistemas internos de negocio.
```

El balance de tono deseado:

```text
80% confianza técnica / negocio serio
20% impacto visual moderno / wow
```

---

## 5. Escalera de soluciones

El sitio público debe comunicar y soportar una escalera de soluciones progresiva.

### Nivel 1 — Sistemas operativos bien diseñados

Sin IA requerida.

Ejemplos:

- dashboards
- CRM interno
- paneles operativos
- workflows
- clientes / proyectos
- stock
- entregas / ruteo
- soporte
- reservas
- reportes

Promesa:

```text
Tu operación deja de depender de planillas, WhatsApp y memoria humana.
```

### Nivel 2 — Automatización e integración

Ejemplos:

- integraciones por API
- alertas
- reportes automáticos
- sincronización entre sistemas
- creación de tareas
- validación de estados
- automatización de flujos de trabajo

Promesa:

```text
Tus sistemas empiezan a hablar entre sí.
```

### Nivel 3 — Agentes IA en la nube

Agente conectado al contexto real del sistema del cliente: bases de datos, apps, reglas y flujos de trabajo.

Ejemplos:

- planificación de rutas
- programación de entregas
- agendamiento de proveedores
- reportes
- interpretación de datos
- ejecución de acciones dentro de la plataforma

Nota: el costo de suscripción al modelo IA debe ser pagado por el cliente cuando sea requerido.

Promesa:

```text
IA conectada al contexto de tu negocio, no aislada en un chat.
```

### Nivel 4 — IA local en servidor/backend

Mayor inversión, menor latencia, más control, contexto privado, modelo local/server-side conectado al sistema operativo.

Promesa:

```text
Inteligencia operativa dentro de tu propia infraestructura.
```

### Nivel 5 — Hardware + software + IA local + sensores/dispositivos

Tier premium/vanguardia.

Ejemplos:

- IA local
- comandos de voz
- sensores
- dispositivos
- kioscos
- terminales
- impresoras
- lectores
- integraciones IoT
- automatización de procesos físicos

Promesa:

```text
Cuando el proceso no termina en una pantalla, Arkanum conecta software, IA y el mundo físico.
```

## 6. Familia de productos / capacidades

El sitio debe preparar espacio para nombrar capacidades o productos de Arkanum sin sobrevender un catálogo SaaS rígido demasiado pronto.

La arquitectura pública debe usar **nombre comercial suave + explicación clara + valor concreto**. Ningún nombre interno debe aparecer solo, porque para un cliente frío `Core`, `Flow` o `Agents` no significan nada si no se traducen inmediatamente a un beneficio operativo.

### Arkanum Studio como paraguas

**Arkanum Studio** será el paraguas comercial inicial: la casa donde se diseñan, adaptan y combinan las capacidades de Arkanum según el problema de cada empresa.

Arkanum Studio no debe presentarse como una agencia genérica ni como un quinto pilar equivalente. Debe funcionar como el método/casa que combina los pilares.

Posible framing:

```text
En Arkanum Studio diseñamos soluciones a medida combinando software, automatización, IA e integración física según el problema de tu empresa.
```

### Cuatro pilares combinables

```text
Arkanum Core           — plataforma operativa / capa central
Arkanum Flow           — automatización e integraciones
Arkanum Agents         — agentes IA conectados a sistemas, datos y reglas
Arkanum Hardware Layer — software conectado a dispositivos, sensores, terminales o infraestructura física
```

Traducción pública recomendada:

| Pilar | Explicación clara | Valor para el cliente |
|---|---|---|
| Arkanum Core | Plataformas operativas para centralizar procesos, datos, equipos y decisiones. | Ver y controlar la operación desde un solo lugar. |
| Arkanum Flow | Automatización e integración entre áreas, sistemas, APIs y tareas repetitivas. | Hacer que los sistemas trabajen entre sí. |
| Arkanum Agents | Agentes IA conectados a datos, reglas y operaciones reales. | Pedir acciones inteligentes a una IA que entiende el negocio. |
| Arkanum Hardware Layer | Software conectado a sensores, dispositivos, terminales y procesos físicos. | Conectar el mundo físico con el sistema. |

Recomendaciones:

- mostrar Arkanum Studio como paraguas público inicial
- mostrar Arkanum Core como el concepto principal de plataforma
- mostrar Flow, Agents y Hardware Layer como capacidades combinables
- evitar que la oferta parezca un catálogo SaaS fijo
- evitar que la empresa parezca que hace trabajo custom sin estructura
- cada nombre debe ir siempre acompañado de explicación clara y valor concreto

## 7. Principios de referencia de diseño

El rediseño no debe copiar literalmente los sitios de referencia. Debe extraer principios de diseño.

### Linear — Hacer el producto tangible

Arkanum debe mostrar un mini dashboard interactivo / preview de plataforma para que el visitante vea cómo podría verse un sistema a medida.

Posibles elementos:

- mini admin dashboard
- leads / clientes / proyectos / seguimientos
- métricas / alertas / actividad
- interacciones simples: tabs, hover, selección, filtros, cambios de estado

Propósito:

```text
No solo describir el sistema. Mostrarlo.
```

### Stripe — Hacer visible la infraestructura

Arkanum debe mostrar que conecta procesos, sistemas, datos, equipos, automatizaciones y posiblemente dispositivos.

Posibles elementos:

- mapa operativo interactivo
- nodos para ventas, operaciones, stock, soporte, administración, reportes, agentes, dispositivos
- Arkanum como la plataforma/capa central que conecta la operación

Propósito:

```text
Mostrar que Arkanum no construye pantallas aisladas. Construye sistemas conectados.
```

### Refokus — Darle ritmo al contenido

Usar reveal, loading y transiciones para que el sitio se sienta vivo sin convertirse en un juego o una pieza de arte.

Propósito:

- guiar la atención
- revelar jerarquía
- evitar bloques estáticos/genéricos
- hacer que la experiencia se sienta premium e intencional

## 8. Dirección visual

El lenguaje de diseño deseado:

```text
dark premium, pero no plano
gradientes controlados
cards con profundidad real
líneas finas de blueprint / circuito / flujo
micro-interacciones suaves
mini dashboard real
mapa de procesos conectados
copy directo
más espacio en blanco
menos cards genéricas
más escenas visuales
```

Las animaciones deben ser exactas y con propósito. Deben explicar, revelar, conectar o guiar. No deben existir solo para decorar.

El sitio no es un libro, pero tampoco es un juego. Es una experiencia comercial para decisiones de negocio serias y proyectos que pueden empezar pequeños pero crecer hacia implementaciones mucho más grandes.

## 10. Narrativa pública

El cliente llega con una necesidad. Debe irse con la idea de que su empresa puede evolucionar.

Arco narrativo:

```text
Tenés un proceso que ya no puede correr con herramientas improvisadas.
Arkanum diseña y construye el sistema operativo que tu negocio necesita.
Podés empezar con un módulo enfocado y evolucionar hacia una plataforma conectada, automatizada e inteligente.
```

Arkanum Studio debe funcionar como el puente entre necesidad y solución:

```text
No vendemos una plantilla.
No forzamos un producto cerrado.
En Arkanum Studio diseñamos la solución que tu operación necesita, combinando Core, Flow, Agents y Hardware Layer cuando el caso lo requiere.
```

---

## 12. Non-goals para la próxima fase de UX/diseño

- No abrir issues de polish visual sin esta dirección.
- No agregar animaciones solo porque se ven bien.
- No hacer que el sitio parezca un portfolio de arte.
- No sobre-posicionar a Arkanum como empresa de chatbots / wrappers de IA.
- No hacer que toda solución parezca requerir IA.
- No hacer que el sitio público apunte a `/admin`.
- No presentar un catálogo SaaS rígido si la solución es todavía custom/adaptativa.
- No usar nombres como Studio, Core, Flow, Agents o Hardware Layer sin explicación clara y valor concreto.
